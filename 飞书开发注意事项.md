# 飞书多维表格 (Bitable) OpenAPI 避坑指南与开发注意事项

在将“飞书多维表格 (Bitable)”作为应用后段轻量级数据库进行集成调用的过程中，由于其严格的数据校验规则与特定的富文本嵌套结构，很容易导致新手遇到接口报错或数据解析异常。
本指南旨在总结本次“共享日历”项目的飞书对接经验，为了未来其它同样使用飞书的研发项目提供专业的复用指南。

---

## ⚠️ 1. 数据结构强校验陷阱：Filter 参数的“潜规则”

### 踩坑回顾：
在调用**搜寻记录接口 (`search`)**时，如果您仅仅提供了 `conditions`（查询条件）却未指定 **`conjunction` (条件关系)**，飞书后台不会默默通过，而是会无情地吐出 HTTP 400 报错并抛出：`field validation failed: filter.conjunction is required`。

### 避坑指南：
*   **凡是带 `filter` 的查询，必须配套 `conjunction`。** 即使您的筛选条件里只有唯一的一个规则 (比如只有一个字段要求匹配 `username == 178`)。
*   `conjunction` 的值必须且只能是：`"and"` （且） 或 `"or"` （或）。

**正确的写法示范：**
```python
search_body = {
    "filter": {
        "conjunction": "and",  # 哪怕只有一个 condition，也必须写上
        "conditions": [
            {
                "field_name": "username",
                "operator": "is",
                "value": ["wangyang"]  # 这里的值也必须是一个 list 数组包裹
            }
        ]
    }
}
```

## ⚠️ 2. 单值字段变成“富文本数组”：返回数据的防沉降处理

### 踩坑回顾：
飞书多维表格里的“文本” (Text) 字段格式非常强大，它可以容纳富文本样式甚至 @ 人的内容。这就意味着，当你通过 API 获取刚才写入的例如 `"nickname": "王小明"` 的字段时，API 给你的返回值是一组列表结构，而非扁平字符串！
错误获取：`fields.get('nickname')` ->  拿到的是 `[{"type": "text", "text": "王小明"}]`，这给前端页面展示和二次判断带来了极大的灾难。

### 避坑指南：
*   **后端必须作为数据“护城河”进行类型降级。** 在向前端下发这些数据之前，您始终需要在服务端通过判断类型，从飞书复杂的“富文本对象列表”中抽取纯文本。

**正确的解析范式：**
```python
nickname = fields.get("nickname", "")
# 如果该值被飞书 API 以序列化格式返回（比如因为他是富文本当成数组处理）
if isinstance(nickname, list):
    # 抽取数组中第一个对象的 'text' 键，如果内容存在的话
    nickname = nickname[0].get("text", "") if nickname else ""
```

## ⚠️ 3. 飞书内建的时间戳问题：毫秒还是格式化字符串？

### 踩坑回顾：
飞书里的“日期”格式，在读写的时候常常造成混乱。如果你存储的字段被定义为“日期类型 (Type 5)”，飞书吐回前端时不仅没有格式化的时区字符串（如 `YYYY-MM-DD`），反而可能吐给你个令人困惑的长串整数——它是**带有毫秒的 UNIX 时间戳**，如果直接在不校验的情况下让前端使用，将渲染错误。

### 避坑指南：
*   **对于纯粹作为标识查询的字符串日期（如 “2026-03-15”）：强烈建议建表时建立为“多行文本 (Type 1)”而非“日期 (Type 5)”。** 这在搜索（例如搜索包含 `2026-03-` 的全月日历）和读写操作上要简单无数倍。
*   **如果坚持使用类型 5：** 在拉取数据时，如果拿到的是 `int/float`，必须在服务端先把它从毫秒时间戳转换出来。

**日期读取防护示例：**
```python
date_val = fields.get("date", "")
# 如果飞书返回的是时间戳
if isinstance(date_val, (int, float)):
    # 需要除以 1000 转回正常的秒级别再去格式化
    date_val = datetime.datetime.fromtimestamp(date_val / 1000).strftime('%Y-%m-%d')
```

## ⚠️ 4. 垃圾数据与主键管理

### 踩坑回顾：
多维表格没有天然的业务级“唯一键约束”在后台生效，如果接口（或本地调试期间）不进行防护拦截，随意 `POST` 可以创造出包含几十条一模一样 `username` 的冗余数据。此外，每一行都有专属生成的随机且无法预测的 `record_id`，若你没有在拉取后暴露它并将其跟数据绑定，后续修改 / 删除请求就无从下手了。

### 避坑指南：
*   **查询后再新增。** 在 `Create` (注册/添加数据) 发生前，一定要预先使用带 `"conjunction"` 的 `search` API 探一探路。 `if result.get("data", {}).get("total", 0) > 0: return fail`，以人工充当唯一键约束。
*   **下穿必备标识。** 下传给页面端的数据体中，必须把本记录原始且关键的 `item.get("record_id")` 也包裹进去。这是日后对该笔数据进行 `PUT` （修改）或 `DELETE`（废弃）追踪的唯一凭证。
